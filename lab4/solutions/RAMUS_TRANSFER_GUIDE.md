# Руководство по переносу IDEF0 диаграмм в Ramus Educational

Данный документ содержит детальное описание структуры всех созданных диаграмм для переноса в Ramus Educational.

## Общая информация о модели

**Проект:** SportRent - Система управления прокатом спортивного оборудования
**Автор:** Студент
**Дата:** 02.11.2025
**Версия:** 1.0

## Иерархия модели

```
A-0  Контекстная диаграмма (создана в Lab 3)
 └── A0  Первый уровень декомпозиции (6 блоков)
      └── A3  Второй уровень декомпозиции (5 блоков)
```

---

## Диаграмма A0 - Первый уровень декомпозиции

### Метаданные диаграммы
- **Узел:** A0
- **Название:** Управление прокатом спортивного оборудования SportRent
- **Родитель:** A-0 (контекстная диаграмма)
- **Количество блоков:** 6

---

### Функциональные блоки

#### Блок A1: "Управлять аутентификацией и профилями клиентов"
**Назначение:** Регистрация, аутентификация пользователей и управление профилями клиентов

**Входы:**
- I2: Персональные данные клиента (от внешнего источника)

**Управление:**
- C1: ФЗ-152 о персональных данных
- C2: Политика компании

**Выходы:**
- Профиль клиента → A3

**Механизмы:**
- M1: Клиенты
- M3: Администраторы
- M5: СУБД PostgreSQL

---

#### Блок A2: "Управлять каталогом оборудования"
**Назначение:** Ведение каталога спортивного оборудования, управление характеристиками и ценами

**Входы:**
- I3: Данные об оборудовании (от внешнего источника)

**Управление:**
- C2: Политика компании
- C3: ГОСТ 34.602-2020

**Выходы:**
- Информация о доступности → A3

**Механизмы:**
- M2: Менеджеры
- M5: СУБД PostgreSQL

---

#### Блок A3: "Обрабатывать бронирование и заказы"
**Назначение:** Обработка заявок на аренду, формирование бронирований и заказов

**Входы:**
- I1: Заявка клиента на аренду (от внешнего источника)
- Профиль клиента (от A1)
- Информация о доступности (от A2)

**Управление:**
- C2: Политика компании
- C4: Алгоритмы обработки заказов

**Выходы:**
- O1: Договор аренды (внешний выход)
- Данные для оплаты → A4
- Подтвержденный заказ → A5
- Данные о заказах → A6

**Механизмы:**
- M1: Клиенты
- M2: Менеджеры
- M5: СУБД PostgreSQL

**Примечание:** Этот блок декомпозируется на втором уровне (A3 → A31-A35)

---

#### Блок A4: "Обрабатывать платежи"
**Назначение:** Проведение финансовых операций, обработка платежей клиентов

**Входы:**
- I4: Платежная информация (от внешнего источника)
- Данные для оплаты (от A3)

**Управление:**
- C2: Политика компании

**Выходы:**
- O2: Квитанция об оплате (внешний выход)
- Подтверждение оплаты → A5
- Данные о платежах → A6

**Механизмы:**
- M5: СУБД PostgreSQL
- M6: Платежная система

---

#### Блок A5: "Управлять складскими операциями"
**Назначение:** Выдача и возврат оборудования, управление складским учетом

**Входы:**
- Подтвержденный заказ (от A3)
- Подтверждение оплаты (от A4)

**Управление:**
- C2: Политика компании

**Выходы:**
- O3: Акт приема-передачи (внешний выход)
- O5: Информация о статусе (внешний выход)
- Данные о выдаче/возврате → A6

**Механизмы:**
- M4: Складской персонал
- M5: СУБД PostgreSQL

---

#### Блок A6: "Формировать отчетность"
**Назначение:** Создание аналитических отчетов и статистики по работе системы

**Входы:**
- Данные о заказах (от A3)
- Данные о платежах (от A4)
- Данные о выдаче/возврате (от A5)

**Управление:**
- C3: ГОСТ 34.602-2020

**Выходы:**
- O4: Отчеты и аналитика (внешний выход)

**Механизмы:**
- M2: Менеджеры
- M3: Администраторы
- M5: СУБД PostgreSQL

---

### Рекомендации по размещению блоков в Ramus

При создании диаграммы A0 в Ramus Educational рекомендуется следующее расположение блоков:

```
Строка 1 (верх):     A1            A2
Строка 2 (середина): A3            A4
Строка 3 (низ):      A5            A6
```

Это обеспечит логичный поток данных слева направо и сверху вниз, минимизирует пересечения стрелок.

## Диаграмма A3 - Второй уровень декомпозиции

### Метаданные диаграммы
- **Узел:** A3
- **Название:** Обработка бронирования и заказов
- **Родитель:** A0 (блок A3)
- **Количество блоков:** 5

---

### Функциональные блоки

#### Блок A31: "Проверять доступность оборудования"
**Назначение:** Проверка наличия запрошенного оборудования в указанные сроки

**Входы:**
- I1: Заявка клиента на аренду (от внешнего источника, из родительского уровня)
- Доступность: Информация о доступности (от A2 родительского уровня)

**Управление:**
- C4: Алгоритмы обработки заказов

**Выходы:**
- Результат проверки → A32

**Механизмы:**
- M5: СУБД PostgreSQL

---

#### Блок A32: "Формировать бронирование"
**Назначение:** Создание предварительного бронирования оборудования

**Входы:**
- Профиль: Профиль клиента (от A1 родительского уровня)
- Результат проверки (от A31)

**Управление:**
- C4: Алгоритмы обработки заказов

**Выходы:**
- Данные бронирования → A33

**Механизмы:**
- M1: Клиенты
- M5: СУБД PostgreSQL

---

#### Блок A33: "Оформлять договор аренды"
**Назначение:** Формирование договора аренды на основе бронирования

**Входы:**
- Данные бронирования (от A32)
- Статус оплаты (от A35) - обратная связь

**Управление:**
- C2: Политика компании
- C4: Алгоритмы обработки заказов

**Выходы:**
- O1: Договор аренды (внешний выход, к клиенту)
- Проект договора → A34

**Механизмы:**
- M2: Менеджеры
- M5: СУБД PostgreSQL

---

#### Блок A34: "Передавать заказ на оплату"
**Назначение:** Подготовка и передача данных в платежную систему

**Входы:**
- Проект договора (от A33)

**Управление:**
- C2: Политика компании

**Выходы:**
- К A4: Данные для оплаты (к блоку A4 родительского уровня)
- Запрос на подтверждение → A35

**Механизмы:**
- M5: СУБД PostgreSQL

---

#### Блок A35: "Подтверждать заказ"
**Назначение:** Финальное подтверждение заказа после оплаты

**Входы:**
- Запрос на подтверждение (от A34)

**Управление:**
- C2: Политика компании

**Выходы:**
- К A5: Подтвержденный заказ (к блоку A5 родительского уровня)
- К A6: Данные о заказах (к блоку A6 родительского уровня)
- Статус оплаты → A33 (обратная связь)

**Механизмы:**
- M2: Менеджеры
- M5: СУБД PostgreSQL

---

### Рекомендации по размещению блоков в Ramus

При создании диаграммы A3 в Ramus Educational рекомендуется следующее расположение блоков:

```
Строка 1 (верх):     A31 → A32 → A33
Строка 2 (низ):              A34 → A35
```

Блоки A31, A32, A33 располагаются в верхнем ряду слева направо.
Блоки A34 и A35 располагаются в нижнем ряду.
Это создает естественный поток процесса сверху вниз и слева направо.

## Общие рекомендации по работе в Ramus Educational

### Создание новой модели
1. Запустите Ramus Educational
2. Создайте новый проект: File → New
3. Сохраните проект с именем: `SportRent_IDEF0.rsf`

### Создание контекстной диаграммы A-0
1. В дереве модели выберите корневой узел
2. Создайте контекстную диаграмму (уже создана в Lab 3)
3. Убедитесь, что диаграмма имеет номер A-0

### Создание уровня A0
1. Выберите блок A0 на контекстной диаграмме
2. Нажмите кнопку "Перейти к дочерней диаграмме" (стрелка вниз)
3. Выберите "Простой шаблон" (Simple template)
4. Укажите количество блоков: 6
5. Создайте блоки A1-A6 согласно описанию выше
6. Присоедините стрелки от границ диаграммы к соответствующим блокам
7. Создайте внутренние стрелки между блоками

### Создание уровня A3
1. Вернитесь на диаграмму A0
2. Выберите блок A3
3. Нажмите кнопку "Перейти к дочерней диаграмме"
4. Выберите "Простой шаблон"
5. Укажите количество блоков: 5
6. Создайте блоки A31-A35 согласно описанию выше
7. Присоедините стрелки и создайте внутренние связи

### Работа со стрелками
- Для присоединения стрелки: наведите курсор на начало/конец стрелки до выделения, нажмите и перетащите к нужной стороне блока
- Для создания внутренней стрелки: переключитесь в режим создания стрелок, нажмите на сторону блока-источника, подведите к стороне блока-приемника
- Для ветвления: создайте первую ветвь от источника до первого получателя, затем нажмите на линию и протяните к другим получателям
- Избегайте туннелирования стрелок (квадратные скобки) - используйте только при крайней необходимости

### Добавление описаний
1. Для каждого блока: правой кнопкой → Properties → Description
2. Для каждой стрелки: правой кнопкой → Properties → Description
3. Используйте описания из этого документа

### Изменение статуса диаграммы
1. После завершения диаграммы: Diagram → Diagram Properties → Status
2. Измените статус с "Draft" на "Working" или "Recommended"

### Экспорт диаграмм
1. File → Export → Image
2. Выберите формат PNG
3. Экспортируйте все три уровня (A-0, A0, A3)
4. Сохраните в `lab4/solutions/diagrams/`
